实体-联系模型

实体-联系模型 ( Entity - Relationship / E - R 模型)是一种语义建模方法——以意义/语意为核心来描述现实世界事物和它们之间关系的方法——力图描述真实世界的含义。

数据库建模则主要包含两方面：

  • 实体的联系

  • 实体集合

概念

  • 实体指的是现实世界中可区别于其他对象的事物或对象。

  • 属性描述实体的性质或特征。实体的每个属性都有一个值。

  • 实体集则表示相同类型或性质的实体集合。

  • 联系指的是多个实体间的相互关联。

  • 联系集则是同类联系的集合。

属性

属性分为简单属性和复合属性。

  • 简单属性不能再分为更小的部分。

    • 如 StudentID 。

  • 复合属性可以再分，由其他属性组合构成。

    • 如 Address 。
graph TD

  • 针对特定的实体，属性只能取单一值的叫单值属性。

  • 若属性可以对应多个值的叫多值属性。

  • 若该属性的值可以通过其他属性或派生属性计算或推导得出的叫派生属性。

映射基数

映射基数 (Mapping Cardinalities) 指的是一个实体通过联系集能同时联系实体的数量。

讨论实体集 A 和 B 之间的二元关系集，有以下几种可能：

  • 一对一
  • 一对多
  • 多对一
  • 多对多

参与约束指的是一个实体集中的实体是否“必须”参与到某个关系集中：

  • 全部参与：实体集中的每一个实体都至少参与到联系集中的一个联系。
  • 部分参与：实体集中只有部分实体参与到联系集合的联系中。

设计ER模型

使用实体集还是属性 / 联系集

判断类型问题含义
属性 / 实体集某个信息应该建成一个属性？还是建成一个独立的实体？
联系集 / 实体集某个“中间对象”是两个实体的关系？还是应该单独建成实体？

  • 如果信息简单、不参与其他关系 → 用属性

  • 如果信息复杂、有结构性、可能会被引用或扩展 → 用实体集

  • 如果某个“关系”中本身包含额外信息 / 属性，或是可以单独管理的内容 → 也可以提升为实体集

联系集的元数和布局

联系集的“元数”指的是一个联系集涉及的实体集数量：

类型定义举例
一元联系集只涉及一个实体集（即自己和自己之间自关联）员工管理员工（一个员工是另一个的上司）
二元联系集涉及两个实体集学生选课、员工属于部门
多元联系集涉及三个或以上实体集医生给病人在某时间开处方

布局问题则涉及的是关系本身的属性，不属于参与的某一方，而是属于“联系”本身：

映射基数类型联系属性放置位置示例
一对一放在任意一侧的实体集中（通常选一侧）老师负责教室：责任人姓名
一对多放在“多”一侧的实体集中部门管理多个员工：入职时间
多对多放在联系集本身中学生选课：成绩、评价

强弱实体集

概念定义是否有主码？是否独立存在？
强实体集可以独立存在的实体，有自己的主属性（主码）是是
弱实体集依赖于其他实体才能存在，本身没有完整主码否否

[例] 每个 CUSTOMER 都可以有一个 ACCOUNT ，但后者没有主码，靠 account_no 不一定能区分出彼此，只能依附于 CUSTOMER 。所以后者是一个弱实体集。

扩展ER特性

扩展 ER 特性（Extended ER Features, EER）是在基本 ER 模型的基础上，引入面向对象思想，来表达更复杂的现实世界结构。

特殊化

特殊化是将一个通用实体集划分为多个更具体的子类实体集的过程。自顶向下。

[例]

转换为关系模式

强实体集

直接对应一个关系模式。

弱实体集

和关联的强实体集的主码一同构成一个关系模式。

联系集

可以参考上文联系集的布局：

联系类型转换方式
一对一合并到任一实体中或单独建表
一对多外键加到“多”方
多对多建一个独立关系表，包含两个外键和属性

复合属性

可以为子属性的每一个属性，都建一个单独的属性合并到父属性中。

多值属性

单独为它创建一个模式，包括所属实体集的主码和它自己的属性。

关系模式设计

关系数据库的设计中的核心问题就是——如何处理数据冗余带来的各种问题。数据冗余指的就是相同的数据在数据库中被重复存储了多次。

包括：

  • 更新异常：修改一条记录时，必须修改多个地方，否则数据会不一致。

[例] 学生选课表 Enrollments ：

student_idstudent_namecourse_idcourse_name
1001AliceC101Math
1001AliceC102English

现在 Alice 改名成 “Alicia”，我们就得修改 多行记录中的名字，漏改会导致数据不一致。

  • 插入异常：想插入一条数据时，由于设计的不好，必须插入不必要的信息，否则无法插入。

例如选课表中如果要录入 Physics 课程，但目前没有人选，所以必须要插入垃圾信息。

  • 删除异常：删除一条数据时，意外地删除了不该删除的信息。

如果删除 Alice 选修 English 的信息：

因为表中只有这一门 English 课，删除它就会导致我们失去了所有 English 课程的信息。

那么在设计表时，应该选择更大的表还是更小的表（模式）？

特点问题
更大的模式一个大表，包含多个实体信息冗余多、异常多
更小的模式拆分成多个小表（规范化）查询要连接多个表（JOIN效率）
  • 分解成更小的模式是更好的
  • 但是有些分解会带来代价

模式分解的目标

解释
每个子模式都是“好”的拆出来的每个表都不应该有数据冗余或异常（达到某种范式）
分解是无损连接的（Lossless Join）多个子表 ⋈\Join⋈ 回去，不能丢失原始数据，也不能多出假数据

第一范式 (1NF)

一个关系模式如果其每个属性值都是不可再分的原子值，那么这个关系模式就满足第一范式[1]（ 1NF ）。

也就是说表中的每个单元格只能存一个值，不能是列表、集合、数组这类可以再拆分的东西。

函数依赖

在关系模式 RRR 中，如果两个元组在属性 AAA 上的值相同，那么它们在属性 BBB 上的值也必须相同——这个依赖关系叫做：BBB 函数依赖于 AAA，或记为 A→BA \rightarrow BA→B 。

[例]

学号（A）姓名（B）
1001张三
1002李四
1001张三

我们可以说：学号 -> 姓名 成立，因为只要学号一样，姓名就一定一样（都是张三）。但反过来 姓名 -> 学号 不成立，因为多个学生可能同名。

函数依赖集的闭包

给定函数依赖集 FFF，某个属性集 XXX 的闭包[2] X+X^+X+ 是：在 FFF 的逻辑蕴含下，可以从 XXX 推导出的所有属性集合。

X+X^+X+ 则可以理解为——从X出发，能推理出的所有一定会相同的属性。

[例] 给定关系模式：R(A,B,C,D)R(A, B, C, D)R(A,B,C,D) 和函数依赖集：F={A→B,B→C}F = \{ A \rightarrow B, B \rightarrow C \}F={A→B,B→C} ，那我们来求 A+A^+A+（ AAA 的闭包）：

  1. 开始闭包：A+={A}A^+ = \{A\}A+={A}
  2. A→BA \rightarrow BA→B，加入 BBB ：A+={A,B}A^+ = \{A, B\}A+={A,B}
  3. 以此类推加入 CCC ：A+={A,B,C}A^+ = \{A, B, C\}A+={A,B,C}
  4. 没别的可推了

所以：A+={A,B,C}A^+ = \{A, B, C\}A+={A,B,C}

函数依赖可以有以下的作用：

  • 设计主码：一个候选码就是能唯一决定所有其他属性的属性集（即 候选码 → 所有属性）

  • 做规范化：通过分析函数依赖决定哪些属性属于哪个表

逻辑蕴含

如果函数依赖集 FFF 推导出某个依赖 fff，即所有满足 FFF 的关系也都满足 fff，则称：FFF 逻辑蕴含 fff，记作 F⊨fF \vDash fF⊨f 。

[例] 给定关系模式：R(A,B,C,D)R(A, B, C, D)R(A,B,C,D) 和函数依赖集：F={A→B,B→C}F = \{ A \rightarrow B, B \rightarrow C \}F={A→B,B→C} ，求证：FFF 逻辑蕴含 A→CA \rightarrow CA→C 。

  • 因为 A→BA \rightarrow BA→B
  • 且 B→CB \rightarrow CB→C
  • 所以 A→CA \rightarrow CA→C （通过传递律）

综上所述，F⊨A→CF \vDash A \rightarrow CF⊨A→C，这个过程也叫 推导 或 逻辑蕴含判断。

Armstrong 公理

Armstrong 公理是三条用于函数依赖推导的基本规则，可以用它们来证明某些依赖是否被逻辑蕴含。

规则
自反律如果 B⊆AB \subseteq AB⊆A，则 A→BA \rightarrow BA→B
增广律若 A→BA \rightarrow BA→B，则 AC→BCAC \rightarrow BCAC→BC [3]
传递律若 A→BA \rightarrow BA→B 且 B→CB \rightarrow CB→C，则 A→CA \rightarrow CA→C

证明 X→YX \rightarrow YX→Y 是否能从 FFF 推出的一般方式：

  1. 求 X+X^+X+（在 FFF 下求闭包）
  2. 看 Y⊆X+Y \subseteq X^+Y⊆X+ 是否成立

若成立，则 F⊨X→YF \vDash X \rightarrow YF⊨X→Y 成立

FFF 是一个函数依赖集， FFF 蕴含的所有函数依赖的集合称为 FFF 的闭包，记作 F+F^+F+ 。

[例]

给定关系模式：

R=(A,B,C,G,H,I)R = (A, B, C, G, H, I)R=(A,B,C,G,H,I)

给定函数依赖集：

F={A→B, A→C, CG→H, CG→I, B→H}F = \{ A \rightarrow B,\ A \rightarrow C,\ CG \rightarrow H,\ CG \rightarrow I,\ B \rightarrow H \}F={A→B, A→C, CG→H, CG→I, B→H}

推导一：A→HA \rightarrow HA→H

  • 已知：A→BA \rightarrow BA→B，B→HB \rightarrow HB→H
  • 运用传递律（Transitivity Rule）：

A→B→H⇒A→HA \rightarrow B \rightarrow H \Rightarrow A \rightarrow HA→B→H⇒A→H

推导二：CG→HICG \rightarrow HICG→HI

  • 已知：CG→HCG \rightarrow HCG→H，CG→ICG \rightarrow ICG→I
  • 可组合为：

CG→HICG \rightarrow HICG→HI

也可通过增广推导形式：

  • CG→HCG \rightarrow HCG→H 增广得：CGCG→HCGCGCG \rightarrow HCGCGCG→HCG
  • 已知 HCG→HIHCG \rightarrow HIHCG→HI（通过组合依赖）
  • 运用传递律：

CGCG→HICGCG \rightarrow HICGCG→HI

但由于本例中 CG→HCG \rightarrow HCG→H 和 CG→ICG \rightarrow ICG→I 已知，直接合并即可：

CG→HICG \rightarrow HICG→HI

推导三：AG→IAG \rightarrow IAG→I

  • 已知：A→CA \rightarrow CA→C，因此 AG→CGAG \rightarrow CGAG→CG（增广律）
  • 已知：CG→ICG \rightarrow ICG→I
  • 运用传递律：

AG→CG→I⇒AG→IAG \rightarrow CG \rightarrow I \Rightarrow AG \rightarrow IAG→CG→I⇒AG→I

另外还有三条拓展规则：

规则解释
合并律若 α→β\alpha\rightarrow\beta \quadα→β 且 α→γ\quad \alpha \rightarrow \gammaα→γ ，则有 $\alpha \rightarrow \beta \gamma $如果 α\alphaα 可以分别决定 β\betaβ 和 γ\gammaγ，那么它就可以同时决定它们两个。
分解律若 α→βγ\alpha\rightarrow\beta\gammaα→βγ 则 α→β\alpha \rightarrow\beta\quadα→β 且 α→γ\quad \alpha \rightarrow \gammaα→γ如果 α\alphaα 可以决定 β\betaβ 和 γ\gammaγ，那它也可以单独决定每一个。
伪传递律若 $\alpha \rightarrow \beta \quad $ 且 γβ→δ\quad \gamma\beta \rightarrow \deltaγβ→δ则 αγ→δ\alpha\gamma \rightarrow \deltaαγ→δ如果 α\alphaα 能决定 β\betaβ，而 γ\gammaγ 和 β\betaβ 能决定 δ\deltaδ，那么 α\alphaα 和 γ\gammaγ 就能决定 δ\deltaδ。

[例]

给定关系模式：

R=(A,B,C,G,H,I)R = (A, B, C, G, H, I)R=(A,B,C,G,H,I)

函数依赖集：

F={A→B, A→C, CG→H, CG→I, B→H}F = \{ A \rightarrow B,\ A \rightarrow C,\ CG \rightarrow H,\ CG \rightarrow I,\ B \rightarrow H \}F={A→B, A→C, CG→H, CG→I, B→H}

推导 1：A→HA \rightarrow HA→H

  • 已知：A→BA \rightarrow BA→B，B→HB \rightarrow HB→H
  • 运用传递律：

A→B→H⇒A→HA \rightarrow B \rightarrow H \Rightarrow A \rightarrow HA→B→H⇒A→H

推导 2：CG→HICG \rightarrow HICG→HI

  • 已知：CG→HCG \rightarrow HCG→H，CG→ICG \rightarrow ICG→I
  • 运用合并律：

CG→H, CG→I⇒CG→HICG \rightarrow H,\ CG \rightarrow I \Rightarrow CG \rightarrow HICG→H, CG→I⇒CG→HI

推导 3：AG→IAG \rightarrow IAG→I

  • 已知：A→CA \rightarrow CA→C
  • 构造 γ=G\gamma = Gγ=G，β=C\beta = Cβ=C
  • 得到：AG→CGAG \rightarrow CGAG→CG（增广律）
  • 已知：CG→ICG \rightarrow ICG→I
  • 运用伪传递律：

A→C, G⋅C→I⇒AG→IA \rightarrow C,\ G \cdot C \rightarrow I \Rightarrow AG \rightarrow IA→C, G⋅C→I⇒AG→I

Boyce-Codd 范式（BCNF）

超码

设关系模式为 RRR，属性集 K⊆RK \subseteq RK⊆R。若满足：

K→RK \rightarrow RK→R

则称 KKK 是 RRR 的一个超码。用 KKK 的值可以唯一标识关系中的每一个元组。

候选码

KKK 是关系模式 RRR 的一个候选码，当且仅当：

  1. K→RK \rightarrow RK→R （即 KKK 是超码）
  2. KKK 是最小的（不可再简化），即不存在真子集 A⊂KA \subset KA⊂K 满足 A→RA \rightarrow RA→R

平凡函数依赖

若函数依赖 A→BA \rightarrow BA→B 中，B⊆AB \subseteq AB⊆A，则称该依赖为平凡的（Trivial）。

[例] 设关系模式 R=(学号,姓名,班级)R = (学号, 姓名, 班级)R=(学号,姓名,班级)，其中有函数依赖：

  • 学号 -> 姓名

这个依赖是不平凡的，因为：

  • 学号 是决定因素
  • 姓名 不是 学号 的子集
  • 所以这个依赖是非平凡的（不平凡）

而学号, 姓名 -> 姓名 就是平凡的，因为右边的 姓名 是左边的子集。

关系模式 RRR 属于 BCNF，当且仅当对于每一个非平凡函数依赖：

α→β\alpha \rightarrow \betaα→β

若该依赖在 RRR 上成立，则：

α 是 R 的超码\alpha \text{ 是 } R \text{ 的超码}α 是 R 的超码

即：BCNF 不允许任何非超码属性决定其他属性，除非它是候选码或超码。

有一说一，全是废话。

BCNF 分解

BCNF 范式要求——对于关系 RRR 中的每一个非平凡函数依赖 α→β\alpha\rightarrow\betaα→β ，必须满足 α\alphaα 是 RRR 的候选码。如果一个关系模式违背了 BCNF 范式，那就要将它分解。

[例] 对于关系 Employee(ID, Name, Dept, Manager) ，它存在依赖：

  • ID -> Name, Dept, Manager

  • Dept -> Manager

根据候选码的概念（能唯一标识所有属性的最小属性集合），可以看出来：

  • EID 能唯一标识员工 → 能推出全部信息，是候选码。

  • Dept 不能唯一标识员工，只能推出 Manager ，不是候选码。

但是发现：

  • Dept -> Manager 中，Dept 不是候选码 → 不符合 BCNF
  • 因此需要 BCNF 分解 为：
    1. DeptManager(Dept, Manager) — 反映每个部门的管理者
    2. Employee(EID, Name, Dept) — 员工的基本信息

[例] 对于关系 Student(SID, Email, Name) ，存在依赖：

  • SID -> Email, Name（学号能确定邮箱和姓名）
  • Email -> SID, Name（邮箱也能确定学号和姓名）

所以两者都是候选码：

函数依赖是平凡的吗？是候选码？满足 BCNF 吗？
SID → Email, Name否是是
Email → SID, Name否是是

这个例子是符合 BCNF 的，所以不需要分解。

BCNF 分解的流程如下：

  1. 找出关系中的所有候选码
  2. 检查所有依赖 X→YX\rightarrow YX→Y 中，是否 XXX 是候选码
  3. 若有违反 BCNF 的依赖，进行分解
  4. 构造两个子关系：R1=X∪YR_1=X\cup YR1​=X∪Y，R2=R−Y+XR_2=R-Y+XR2​=R−Y+X （注意双方都保留了 XXX ）
  5. 递归处理子关系，直到都满足 BCNF

通常，BCNF 分解是无损的分解[4]。因为在 BCNF 分解中，每次分解都会保证一个关键属性（候选码的某一部分）出现在分解后的关系中。

对于任何违反 BCNF 的关系，我们通过将包含所有依赖属性的候选码部分分解成两个新的关系来保证数据的完整性，并且通过它们的自然连接能恢复出原来的关系。

▶
补充：F+ 在分解过程中的作用

在数据库范式分解过程中，关系依赖闭包 ( F^+ ) 起到了非常重要的作用。具体来说，( F^+ ) 是函数依赖集 ( F ) 的闭包，它包含了所有由 ( F ) 推导出来的函数依赖。理解 ( F^+ ) 在分解过程中的作用，可以帮助我们保证分解后的子关系不仅满足目标范式（如 BCNF 或 3NF），而且能够确保数据的一致性和无损性。

  1. 检查分解后的关系模式的完整性
    在对一个关系模式 ( RRR ) 进行分解时，我们要确保分解后的各个子关系能够保持原始关系的所有依赖。此时，( F^+ ) 就会派上用场。通过计算依赖闭包 ( F^+ )，我们可以确认分解后是否会丢失任何依赖关系，或者是否存在不必要的冗余依赖。
  2. 帮助判断分解是否符合目标范式
    分解过程中的一个关键目标是确保分解后的子关系满足所要求的范式（如 BCNF 或 3NF）。使用 ( F^+ )，我们可以检查每个子关系是否符合目标范式的定义。具体地说，我们可以验证：
    • 如果一个子关系的函数依赖不满足范式要求（如 BCNF 中要求左边是候选码），我们可以通过计算闭包 ( F^+ ) 来识别问题。
  3. 确定候选码和分解后的依赖关系
    在分解过程中，我们需要确保每个子关系的候选码是可以恢复的，而不是丢失。在使用 ( F^+ ) 时，我们可以：
    • 通过计算候选码的闭包，确认每个候选码是否能够唯一标识分解后的子关系。
    • 确保在分解后的关系模式中，候选码不丢失，并且能够通过自然连接（join）恢复出原始关系。
  4. 防止生成有损分解
    如果不考虑 ( F^+ )，可能会进行有损分解。通过使用闭包，我们可以确保分解是无损的。因为 ( F^+ ) 能够提供关于属性之间所有可能的依赖信息，如果我们在分解过程中考虑到所有闭包中的依赖关系，我们就能避免因遗漏某些依赖关系而导致有损分解的情况。

第三范式（3NF）

保持依赖

设：

  • 关系模式为 RRR，函数依赖集为 FFF
  • 分解后的关系为 R1,R2,…,RnR_1, R_2, \dots, R_nR1​,R2​,…,Rn​
  • FiF_iFi​ 是 FFF 在每个 RiR_iRi​ 上的投影

如果：

(F1∪F2∪⋯∪Fn)+=F+(F_1 \cup F_2 \cup \dots \cup F_n)^+ = F^+(F1​∪F2​∪⋯∪Fn​)+=F+

则称该分解是保持依赖的。

[例] 保持依赖的分解

对于关系模式 R=(A,B,C)R = (A, B, C)R=(A,B,C) ，和函数依赖集 F={A→B,B→C}F = \{ A \rightarrow B, B \rightarrow C \}F={A→B,B→C}

候选码是 AAA ，这个关系模式不符合BCNF，因为B→CB \rightarrow CB→C中BBB不是超码。

我们将其分解为：

  • R1=(A,B)R_1 = (A, B)R1​=(A,B)
  • R2=(B,C)R_2 = (B, C)R2​=(B,C)

对应的函数依赖：

  • 在R1R_1R1​上：A→BA \rightarrow BA→B
  • 在R2R_2R2​上：B→CB \rightarrow CB→C

这两个依赖可以直接在两个表中检查，而且：

{A→B,B→C}+=F+\{ A \rightarrow B, B \rightarrow C \}^+ = F^+{A→B,B→C}+=F+

因此，这个分解是：

  • 无损的（满足连接恢复原始关系）
  • 保持依赖的

仍然使用关系R=(A,B,C)R = (A, B, C)R=(A,B,C)，函数依赖集F={A→B,B→C}F = \{ A \rightarrow B, B \rightarrow C \}F={A→B,B→C}

我们尝试另一个分解：

  • R1=(A,C)R_1 = (A, C)R1​=(A,C)
  • R2=(A,B)R_2 = (A, B)R2​=(A,B)

现在：

  • 在R1R_1R1​中，无法检测B→CB \rightarrow CB→C
  • 在R2R_2R2​中也无法检测B→CB \rightarrow CB→C

因此，B→CB \rightarrow CB→C这个依赖丢失了，不能直接验证，只有在 join后才能恢复。

这就属于：不保持依赖的分解。

判定保持依赖有两个方法：

  • 标准做法，基于闭包
  • 简化做法，基于传播法

[例]

对于关系模式：

R=(A,B,C)R = (A, B, C)R=(A,B,C)

和函数依赖集：

F={A→B,B→C}F = \{A \to B, B \to C\}F={A→B,B→C}

假设分解方案为：

R1=(A,B),R2=(A,C)R_1 = (A, B), \quad R_2 = (A, C)R1​=(A,B),R2​=(A,C)

方法一：闭包法

  1. 步骤 1：求 F+F^+F+

F+={A→B,B→C,A→C}F^+ = \{A \to B, B \to C, A \to C\}F+={A→B,B→C,A→C}

  1. 步骤 2：求每个 FiF_iFi​

  • 在 R1=(A,B)R_1 = (A, B)R1​=(A,B) 上：

    F1={A→B}F_1 = \{A \to B\}F1​={A→B}

  • 在 R2=(A,C)R_2 = (A, C)R2​=(A,C) 上：

    F2={A→C}F_2 = \{A \to C\}F2​={A→C}

  1. 步骤 3：合并 FiF_iFi​

F′=F1∪F2={A→B,A→C}F' = F_1 \cup F_2 = \{A \to B, A \to C\}F′=F1​∪F2​={A→B,A→C}

  1. 步骤 4：计算 F′+F'^+F′+

F′+={A→B,A→C}F'^+ = \{A \to B, A \to C\}F′+={A→B,A→C}

  1. 步骤 5：比较 F+F^+F+ 与 F′+F'^+F′+

F+≠F′+⇒不保持依赖（缺少 B→C）F^+ \neq F'^+ \quad \Rightarrow \text{不保持依赖（缺少 } B \to C\text{）}F+=F′+⇒不保持依赖（缺少 B→C）

方法二：简易传播法

  1. 对 A→BA \to BA→B
  • result={A}\text{result} = \{A\}result={A}
  • 在 R1R_1R1​: A→BA \to BA→B ⟹ result={A,B}result = \{A, B\}result={A,B}
  • 在 R2R_2R2​: 没有变化
  • 故 B⊆resultB \subseteq \text{result}B⊆result
  1. 对 B→CB \to CB→C
  • result={B}\text{result}= \{B\}result={B}
  • 在 R1R_1R1​: 无法推出 CCC
  • 在 R2R_2R2​: B∉R2B \notin R_2B∈/R2​，无法操作
  • 故 C⊈resultC \not\subseteq \text{result}C⊆result
  1. 对 A→CA \to CA→C
  • result={A}\text{result}= \{A\}result={A}
  • 在 R1R_1R1​: 无变化
  • 在 R2R_2R2​: A→CA \to CA→C ⟹ result={A,C}\text{result}= \{A, C\}result={A,C}
  • C⊆resultC \subseteq \text{result}C⊆result

结论：

  • 闭包法判断：不保持依赖（丢失 B→CB \to CB→C）
  • 简易传播法判断：不保持依赖（B→CB \to CB→C 无法恢复）

BCNF分解不一定能做到保持依赖：

  • BCNF 强调的是消除所有非平凡的函数依赖中，左边不是超码的情况。
  • 但在分解过程中，可能无法将所有函数依赖保留在某一个分解后的子关系中，因此部分依赖“丢失”，即分解不保持依赖。

[例]

对于 R=(A,B,C)R = (A, B, C)R=(A,B,C)，F=A→B,B→CF = {A \to B, B \to C}F=A→B,B→C，候选码是 AAA ，但 B→CB \to CB→C 左边 BBB 不是候选码，违反 BCNF ，所以需要分解。

分解方案一（保持依赖的）：

  • R1=(A,B)R_1 = (A, B)R1​=(A,B)，R2=(B,C)R_2 = (B, C)R2​=(B,C)
  • 无损连接成立（R1∩R2=B,B→CR_1 \cap R_2 = {B}, B \to CR1​∩R2​=B,B→C 存在于 R2R_2R2​）
  • 所有依赖 A→B,B→CA \to B, B \to CA→B,B→C 均能在子关系中找到
  • 保持依赖

分解方案二（不保持依赖的）：

  • R1=(A,B)R_1 = (A, B)R1​=(A,B)，R2=(A,C)R_2 = (A, C)R2​=(A,C)
  • B→CB \to CB→C 在任何一个分解中都 不完整存在
  • 所以无法检测出 B→CB \to CB→C
  • 不保持依赖

第三范式

如果一个关系模式 RRR 满足函数依赖集 FFF，如果 F+F^+F+ 中的每个函数依赖 α→β\alpha \to \betaα→β 都满足下列至少一个条件，则 RRR 是 第三范式（3NF） 的：

  1. α→β\alpha \to \betaα→β 是 平凡函数依赖（即 β⊆α\beta \subseteq \alphaβ⊆α）
  2. α\alphaα 是 RRR 的 超码
  3. β−α\beta - \alphaβ−α 中的每个属性都包含在某个 候选码 中（可以是不同的候选码）
标准要求依赖左部对右部的要求
BCNF必须是超码无要求
3NF不是超码也可只要右部在某候选码中

所以，比起 BCNF ，3NF 更宽松，允许有用但不规范的依赖存在，可以保持依赖。

设关系 R=(A,B,C)R = (A, B, C)R=(A,B,C)，函数依赖 F=A→B,B→CF = {A \to B, B \to C}F=A→B,B→C，候选码为 AAA。

  • A→BA \to BA→B：左边 AAA 是候选码，满足条件 2
  • B→CB \to CB→C：BBB 不是超码，也不是候选码，但 CCC 在候选码 AAA 中不出现

→ 所以 RRR 不满足 3NF

如果改为：

  • 候选码是 AAA 且 CCC 属于候选码中
  • 或 BBB 是候选码之一
    → 则可能属于 3NF

正则覆盖

函数依赖集 FFF 的正则覆盖 FcF_cFc​ 是一个与 FFF 等价（闭包相同） 的最小集合，去除了所有冗余的部分：

  1. 如果从某个依赖中移除一个属性，结果依赖集闭包不变 → 这个属性是无关属性
  2. 如果整个依赖可以由其他依赖推导出来 → 是冗余依赖

所以，正则覆盖满足：

  • FFF 逻辑蕴含 FcF_cFc​ 中的所有依赖
  • FcF_cFc​ 逻辑蕴含 FFF 中的所有依赖
  • FcF_cFc​ 中的所有依赖都不包含无关属性
  • FcF_cFc​ 中的函数依赖，左半部分都是唯一的

正则覆盖不是唯一的。

  1. “范”：模范、规范、标准。“式”：形式、格式。所以在这里范式就是用来规范 关系型数据库表结构设计 的一系列规则和标准 。 ↩
  2. “闭包”这个词就是“闭合”+“包含”的意味，也就是给定一组起点，所有能“推导”出来的东西，最终形成一个完整的、封闭的集合。 ↩
  3. ACACAC 等效于 A∪CA\cup CA∪C。增广律其实是表达：如果你原来知道 AAA 就能唯一决定 BBB， 那你多知道点信息 （也就是 CCC ）也没问题， 你还是能决定 BBB（甚至带上 CCC 一起决定）。 ↩
  4. 有损分解是指分解后的子关系无法通过自然连接恢复出原始的关系，意味着数据在分解过程中可能会丢失，或出现不一致的结果。例如关系 R(A,B,C,D)R(A, B, C, D)R(A,B,C,D) ，存在依赖 A→BA \to BA→B 和 C→DC \to DC→D。如果将它们分解成 R1(A,B)R_1(A, B)R1​(A,B) 和 R2(C,D)R_2(C, D)R2​(C,D) 。如果通过自然连接 R1R_1R1​ 和 R2R_2R2​，我们无法恢复原始关系，因为没有交集属性，所以这是有损分解。 ↩
